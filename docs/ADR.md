# Architecture Decision Records

## 철학
MVP 속도 최우선. 외부 의존성 최소화. 작동하는 최소 구현을 선택한다. 단, API 키 보안만은 타협하지 않는다.

---

### ADR-001: Next.js 15 App Router 선택
**결정**: Next.js 15 + TypeScript(strict) + Tailwind CSS로 구현한다.
**이유**: API Route로 서버 사이드 시크릿(API 키)을 안전히 격리하면서 동일 프로젝트에서 UI까지 한 번에 처리 가능. 별도 백엔드 불필요.
**트레이드오프**: 단일 HTML/바닐라 JS보다 셋업·빌드 체계가 무겁다.

### ADR-002: YouTube API 키 서버 사이드 격리
**결정**: `YOUTUBE_API_KEY`는 `.env.local`에 저장하고 API Route(`/api/recommend`)와 `services/youtube.ts`에서만 사용한다. 클라이언트 번들에 절대 포함하지 않는다(`NEXT_PUBLIC_` 접두사 금지).
**이유**: API 키가 클라이언트에 노출되면 도용·쿼터 소진 위험. 서버 경유가 유일한 안전 경로.
**트레이드오프**: 모든 외부 호출이 서버 왕복을 거쳐야 한다(MVP 규모에선 무시 가능).

### ADR-003: "최근 + 조회수" 단순 추천 로직 채택
**결정**: 고정 키워드(아이돌 챌린지/댄스챌린지)로 검색하고, 최근 업로드(기본 30일) 중 조회수 내림차순으로 정렬해 상위 N개를 추천한다.
**이유**: 구현이 단순하고 바이럴 의도("핫한 것")에 충분히 부합. 별도 스코어링/ML 불필요.
**트레이드오프**: "증가 속도(velocity)" 기반 트렌드 감지보다 정밀도는 낮다. 오래된 고조회수 영상은 기간 필터로 배제해 보완.

### ADR-004: 테스트는 서비스 레이어로 한정
**결정**: 단위 테스트는 `services/youtube.ts`의 정렬/파싱 로직에 대해서만 작성한다(fetch는 mock). vitest 사용.
**이유**: 외부 API 응답을 가공·정렬하는 부분이 유일한 비자명 로직. MVP 속도를 위해 UI/라우트 테스트는 생략.
**트레이드오프**: UI 회귀는 수동 검증에 의존.
